## 你的角色

你是一位机器人遥操作(teleoperation)与全身逆运动学(whole-body IK)方向的资深系统架构师。我会给你一份现有系统的详细架构报告(`current_architecture.md`),它描述了一套 "Apple Vision Pro → 重定向 → Pink/Pinocchio IK → MuJoCo 仿真" 的上半身整体遥操系统,以及把它接到真机(Astribot S1)的官方 ROS2 接口调研。

我正准备**重构这套系统**,目标是**从 MuJoCo 仿真走向真机测试**。请基于报告给出**具体、可落地的重构与架构建议**。

## 背景与硬约束(重构必须遵守)

请特别注意以下三条,它们是我已经做出的决策,**不要建议我推翻**,而是围绕它们给方案:

### 1. 我用自己的 IK 逻辑,不走机器人自带的 WBC
我决定继续用**自研的 Pink/Pinocchio IK 求解器**输出关节角,然后通过官方的**关节空间命令接口**(`/<部件>/joint_space_command`,消息 `RobotJointController`,详见架构报告)直接下发给真机。
**暂不考虑**走机器人自带 WBC(whole-body controller)那条"下发末端位姿、让机器人本体解 IK"的路径。
→ 请围绕"自研 IK → 关节命令 → 真机"这条主线给建议(节点划分、限速/安全、首帧对齐、坐标系、状态反馈闭环等),不要花篇幅论证该不该用 WBC。

### 2. 系统里有"电脑键盘手动换档"这一交互,重构时必须保留并妥善设计其节点
当前有一套**手动档位覆盖**:操作者在电脑键盘上按数字键(2-9),强制把三个模式切换轴(底盘平移 / 底盘偏航 / 躯干俯仰)锁进 P/N/D 档,定时(5s)后自动释放回自动控制(细节见架构报告 "模式切换与手动换档" 一节)。
→ 请在你的重构架构里**明确这个键盘输入节点/线程的位置**:它如何与 IK 求解节点、与真机命令下发节点解耦?在 ROS2 化之后,键盘信号应该作为独立节点发布话题,还是留在主循环?如何保证它在真机上依然实时、且不会因为换档在关节命令里引入跳变?这是我关心的重点,请详细设计。

### 3. 以下两个模块已被验证效果不好,重构中应舍弃
- **EgoPoser 神经网络先验**(`NeuralPostureTask` + `egoposer/` 子包,由 `neural_posture_cost` 开关):经评估**效益不高**,重构中**删除**,不要在新架构里保留或建议改进它。
- **base_follow / BaseTrackingTask**(`base_track_cost` / `base_follow_scale`,那个"让底盘强行跟随头部运动"的任务):经验证**效果不好**,重构中**删除**。
→ 这两块在当前代码里默认都是关闭(cost=0)状态,但代码结构还在。重构时整块移除即可。请不要围绕它们给建议;如果你对"底盘该如何跟随/是否需要跟随"有想法,可以另提替代思路,但不要复活 base_follow。

### 4. 手/夹爪的重定向要与身体重定向彻底解耦,各自独立求解 IK
当前身体 IK(23-DOF:臂+躯干+颈+底盘,由手腕位姿驱动)与手指重定向(`HandRetargeter → command_fingers`)虽已分成两条路径,但仍耦合在同一主循环里。我要求重构后**两者是独立的求解模块**:
- **身体**由手腕/头位姿驱动整身 IK;**手/夹爪**是一套**独立的手指 IK/重定向**,输入手部关节数据、输出手指关节命令。
- **手部数据缺失时身体照常工作**:实验室订购的灵巧手还没到,所以当没有手部数据(手 pose/keypoints 为 None)时,机器人**应该仍能正常驱动躯干和手臂**(手臂可退化为只跟手腕位姿、或按策略保持),绝不能因为缺手数据就卡住或报错。
- **手部逻辑不是我负责的**:目前代码里手的部分是我随便写的占位。你的设计**必须给手/夹爪重定向留一个干净、明确的接口边界**(输入是什么、输出是什么、如何插入主循环/节点图),方便我的同事之后往里填他自己的具体逻辑(不同灵巧手型号、不同重定向算法),而不需要改动身体 IK 或主循环。请明确这个接口的签名与契约。

### 5. 回放录制时,机器人应先"归位"到起始姿态;并要一个运行时"运动到指定姿态"的功能
两个相关需求:
- **回放对齐**:播放录制的轨迹时,机器人应**先平滑运动到该录制片段起始帧对应的机器人姿态**,到位后再开始回放。避免从当前随机姿态直接跳到录制首帧造成突跳。
- **运行时归位到指定姿态**:增加一个**键盘按键**,按下后让机器人**平滑运动到一个指定的保存姿态**(例如 `my_pose_1.json` 里的姿态)。这需要一个"关节空间轨迹插值到目标姿态"的运动原语(不是瞬间 set qpos,真机上必须限速平滑过渡)。
- **顺带请你考虑**:编辑/保存机器人姿态的功能(当前是独立的 `pose_editor.py` 工具 + `pose_io` 的 JSON 存取,详见架构报告 §8 相关)在重构后应该放在哪?是保留为独立工具,还是集成进主程序的某个模式?运行时"运动到指定姿态"和"保存当前姿态"是否应该成对出现在同一个交互层里?
